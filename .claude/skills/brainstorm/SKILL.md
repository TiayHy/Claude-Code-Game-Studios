1. **解析参数**：获取可选的题材/主题提示（如 `roguelike`、`space survival`、`cozy farming`）。如果是 `open` 或没有参数，则从零开始。同时解析审查模式（一次性读取并缓存，本次运行中所有 gate spawn 共用）：
   1. 如果传入了 `--review [full|lean|solo]` → 使用该值
   2. 否则读取 `production/review-mode.txt` → 使用该值
   3. 否则 → 默认为 `lean`

   完整检查模式参见 `.claude/docs/director-gates.md`。

2. **检查现有概念工作**：
   - 如果存在，读取 `design/gdd/game-concept.md`（继续而非重启）
   - 如果存在，读取 `design/gdd/game-pillars.md`（在已建立的核心支柱上继续）

3. **互动式进行各阶段构思**，在每个阶段向用户提问。不要默默生成所有内容——目标是**协作式探索**，AI 扮演创意促进者，而非人类愿景的替代者。

   在构思的关键决策点使用 `AskUserQuestion`：
   - 约束性偏好问题（题材偏好、规模、团队规模）
   - 概念选择（展示选项后："哪些 2-3 个概念引起了你的共鸣？"）
   - 方向选择（"继续深入、探索更多，还是原型验证？"）
   - 概念细化后的支柱排名
   先在对话文本中写出完整的创意分析，再用 `AskUserQuestion` 以简洁标签捕获决策。

   遵循专业工作室的构思原则：
   - 暂缓评判——探索阶段没有坏点子
   - 鼓励异想天开——跳出框外的思考能激发更好的概念
   - 相互启发——"是的，而且……"而非"但是……"
   - 把约束当作创意燃料——限制往往催生最佳创意
   - 每个阶段设定时间盒——保持节奏，早期不要过度思辨

### 阶段 1：创意发现

先了解这个人，再了解游戏。 conversational 地提出这些问题（而非 checklist 式的）：

**情感锚点**：
- 游戏中有没有哪个时刻真正打动了你、让你兴奋、或让你忘了时间？具体是什么创造那种感觉？
- 在游戏中，你一直想要但从未找到的某种幻想或权力体验是什么？

**品味画像**：
- 你花时间最多的 3 款游戏是什么？是什么让你一直玩下去？
  *（用纯文本形式问这个问题——用户必须能自由输入具体的游戏名。不要把这个问题放在带有预设选项的 AskUserQuestion 里。）*
- 有没有你喜欢的题材？有你回避的题材吗？为什么？
- 你更喜欢挑战你、让你放松、给你讲故事、还是让你表达自己的游戏？*（用 `AskUserQuestion`——约束选项。）*

**实际约束**（在构思前塑造沙盒）。将这些问题打包进一个多标签 `AskUserQuestion`，标签名称如下：
- 标签 "体验"——"你最想让玩家获得什么样的体验？"（挑战 & 精通 / 故事 & 探索 / 表达 & 创作 / 放松 & 心流）
- 标签 "时间线"——"你的实际开发时间线是什么？"（数周 / 数月 / 1-2年 / 多年）
- 标签 "开发水平"——"你的开发历程处于什么阶段？"（第一款游戏 / 有过上线经验 / 专业背景）

严格使用这些标签名称——不要重命名或重复。

**综合**以上答案，形成**创意简报**——3-5 句话概括这个人的情感目标、品味画像和约束条件。朗读简报并确认它捕捉到了他们的意图。

### 阶段 2：概念生成

以创意简报为基础，生成**3 个不同方向**的差异化概念。运用以下构思技巧：

**技巧 1：动词优先设计**
从核心玩家动词开始（建造、战斗、探索、解谜、生存、创作、管理、发现），然后由此向外延伸。动词本身就是游戏。

**技巧 2：混搭法**
将两个意外的元素组合：[题材 A] + [主题 B]。两者之间的张力创造独特的钩子。（例如："农场模拟 + 宇宙恐怖"、"roguelike + 约会模拟"、"城市建造 + 即时战斗"）

**技巧 3：体验优先设计（MDA 逆向）**
从期望的玩家情感出发（MDA 框架中的美学目标：感官、幻想、叙事、挑战、社团、探索、表达、顺从），逆向推导能产生这种情感的动力学和机制。

每个概念都要呈现：
- **暂定标题**
- **电梯演讲**（1-2 句话——必须能通过"10 秒测试"）
- **核心动词**（玩家最常做的单一动作）
- **核心幻想**（情感承诺）
- **独特钩子**（通过"而且还"测试：像 X，**而且还** Y）
- **主要 MDA 美学**（哪种情感占主导？）
- **预估规模**（小 / 中 / 大）
- **为何可行**（一句话说明市场/受众契合度）
- **最大风险**（一句话说明最难回答的问题）

展示全部三个。然后用 `AskUserQuestion` 捕获选择。

**关键**：这必须是一个纯列表调用——不带标签，不带表单字段。严格使用以下结构：

```
AskUserQuestion(
  prompt: "Which concept resonates with you? You can pick one, combine elements, or ask for fresh directions.",
  options: [
    "Concept 1 — [Title]",
    "Concept 2 — [Title]",
    "Concept 3 — [Title]",
    "Combine elements across concepts",
    "Generate fresh directions"
  ]
)
```

这里不要用 `tabs` 字段。`tabs` 表单仅用于多字段输入——在这里使用会导致"Invalid tool parameters"错误。这是一个纯 `prompt` + `options` 调用。

永远不要强迫用户做出选择——让他们自己消化。

### 阶段 3：核心循环设计

针对选定的概念，通过结构化提问来构建核心循环。
核心循环是游戏的心脏——如果它本身不够有趣，再多的内容或打磨也救不了游戏。

**30 秒循环**（即时体验）：

通过 `AskUserQuestion` 提出以下问题——选项从选定概念中推导，不要硬编码：

1. **核心动作手感**——prompt："核心动作的主要感受是什么？"生成 3-4 个符合概念风格和基调的选项，加上一个自由文本出口（`我来描述`）。

2. **关键设计维度**——识别这个概念最重要的设计变量（如世界反应性、节奏、玩家能动性）并就此提问。生成与概念匹配的选项，始终包含自由文本出口。

捕获答案后分析：这个动作本身是否有趣？是什么让它感觉良好？（音频反馈、视觉表现、时机满足感、战术深度？）

**5 分钟循环**（短期目标）：
- 是什么把即时游戏体验组织成循环？
- "再来一回合"/"再来一局"心理从何而来？
- 玩家在这个层面做什么选择？

**会话循环**（30-120 分钟）：
- 一个完整的会话是什么样的？
- 自然停顿点在哪里？
- 是什么"钩子"让他们不玩的时候也会想着这个游戏？

**成长循环**（数天/数周）：
- 玩家如何成长？（力量？知识？选项？故事？）
- 长期目标是什么？什么时候游戏算"完成"？

**玩家动机分析**（基于自我决定理论）：
- **自主性**：玩家有多少有意义的选择？
- **胜任感**：玩家如何感受到自己技能的成长？
- **关联感**：玩家如何感受到联系（角色、其他玩家、或世界）？

### 阶段 4：支柱与边界

游戏支柱是真实 AAA 工作室使用的手段（God of War、Hades、The Last of Us），让成百上千的团队成员在做决策时都指向同一个方向。即使是独立开发者，支柱也能防止范围蔓延、保持愿景清晰。

协作定义**3-5 个支柱**：
- 每个支柱有**名称**和**一句话定义**
- 每个支柱有**设计测试**："如果我们在 X 和 Y 之间争论，这个支柱说我们选择 __"
- 支柱之间应该制造张力——如果所有支柱指向同一个方向，它们就没有发挥足够作用

然后定义**3+ 个反支柱**（这个游戏**不是**什么）：
- 反支柱防止最常见的范围蔓延形式："如果加上……会不会很酷"式的不服务核心愿景的功能
- 表述为："我们**不会**做[某事]，因为它会损害[某支柱]"

**支柱确认**：展示完整支柱集后，使用 `AskUserQuestion`：
- Prompt："这些支柱对你的游戏感觉对吗？"
- Options：`[A] 锁定` / `[B] 重命名或重新定义一个` / `[C] 替换一个支柱` / `[D] 其他`

如果用户选 B、C 或 D，修订后再次使用 `AskUserQuestion`：
- Prompt："支柱已更新。准备锁定了吗？"
- Options：`[A] 锁定` / `[B] 再修订一个支柱` / `[C] 其他`

重复直到用户选择 [A] 锁定。

**审查模式检查**——在生成 CD-PILLARS 和 AD-CONCEPT-VISUAL 之前应用：
- `solo` → 跳过两者。注明："CD-PILLARS 跳过 — Solo 模式。AD-CONCEPT-VISUAL 跳过 — Solo 模式。"进入阶段 5。
- `lean` → 跳过两者（不是 PHASE-GATE）。注明："CD-PILLARS 跳过 — Lean 模式。AD-CONCEPT-VISUAL 跳过 — Lean 模式。"进入阶段 5。
- `full` → 正常生成。

**支柱和反支柱达成一致后，通过 Task 同时并行生成 `creative-director` 和 `art-director`，然后进入阶段 5。不要等一个完成再启动另一个。**

- **`creative-director`** — gate **CD-PILLARS**（`.claude/docs/director-gates.md`）
  传入：完整支柱集含设计测试、反支柱、核心幻想、独特钩子。

- **`art-director`** — gate **AD-CONCEPT-VISUAL**（`.claude/docs/director-gates.md`）
  传入：游戏概念电梯演讲、完整支柱集含设计测试、目标平台（如已知）、用户提到的任何参考游戏或视觉基准。

收集两份评审结果，然后用双标签 `AskUserQuestion` 一起呈现：
- 标签 **"Pillars"**：呈现 creative-director 反馈。选项与标准 CD-PILLARS 处理一致——`原样锁定` / `修订[具体支柱]` / `进一步讨论`。
- 标签 **"视觉锚点"**：呈现 art-director 的 2-3 个命名视觉方向选项。选项：每个命名方向（一个选项一个）+ `混合各方向元素` + `描述我自己的方向`。

用户选择的视觉锚点（命名方向或其自定义描述）存储为**视觉身份锚点**——它将被写入游戏概念文档，并成为艺术圣经的基础。

如果 creative-director 对支柱返回 CONCERNS 或 REJECT，先解决支柱问题再询问视觉锚点选择——视觉方向应从确认的支柱中延伸。

### 阶段 5：玩家类型验证

使用 Bartle 分类法和 Quantic Foundry 动机模型，验证这个游戏实际面向谁：

- **主要玩家类型**：谁会**爱死**这个游戏？（成就者、探索者、社交者、竞争者、创造者、故事讲述者）
- **次要吸引力**：谁可能也会喜欢？
- **不适合谁**：明确谁不会喜欢这个游戏和知道谁会喜欢同样重要
- **市场验证**：有没有成功游戏服务于类似的玩家类型？我们可以从他们的受众规模中学到什么？

### 阶段 6：规模与可行性

让概念脚踏实地：

- **目标平台**：用 `AskUserQuestion`——"你打算在哪些平台发布这个游戏？"
  选项：`PC (Steam / Epic)` / `Mobile (iOS / Android)` / `Console` / `Web / Browser` / `Multiple platforms`
  记录答案——它直接影响引擎推荐，并将传递给 `/setup-engine`。
  注明平台影响（如移动端意味着强烈推荐 Unity；主机意味着 Godot 有局限；网页意味着 Godot 导出干净）。

- **引擎经验**：用 `AskUserQuestion`——"你已经有在用的引擎了吗？"
  选项：`Godot` / `Unity` / `Unreal Engine 5` / `No preference — help me decide`
  - 如果选了某个引擎 → 记录为偏好并继续。不要质疑它。
  - 如果选"无偏好" → 告诉他们："这次会话后运行 `/setup-engine`——它会根据你的概念和目标平台引导你做出完整决策。"不要在这里做推荐。
- **美术管线**：美术风格是什么？劳动强度如何？
- **内容规模**：估算关卡/区域数量、物品数量、游戏时长
- **MVP 定义**：什么是测试"核心循环是否有趣"的绝对最小构建？
- **最大风险**：技术风险、设计风险、市场风险
- **规模层级**：完整愿景是什么？时间用尽时交付什么？

**审查模式检查**——在生成 TD-FEASIBILITY 之前应用：
- `solo` → 跳过。注明："TD-FEASIBILITY 跳过 — Solo 模式。"直接进入规模层级定义。
- `lean` → 跳过（不是 PHASE-GATE）。注明："TD-FEASIBILITY 跳过 — Lean 模式。"直接进入规模层级定义。
- `full` → 正常生成。

**识别最大技术风险后，在定义规模层级之前，通过 Task 使用 gate TD-FEASIBILITY（`.claude/docs/director-gates.md`）生成 `technical-director`。**

传入：核心循环描述、平台目标、引擎选择（或"未决定"）、已识别的技术风险列表。

向用户呈现评估。如果是 HIGH RISK，在定稿前提供重新审视规模的机会。如果是 CONCERNS，注明并继续。

**审查模式检查**——在生成 PR-SCOPE 之前应用：
- `solo` → 跳过。注明："PR-SCOPE 跳过 — Solo 模式。"进入文档生成。
- `lean` → 跳过（不是 PHASE-GATE）。注明："PR-SCOPE 跳过 — Lean 模式。"进入文档生成。
- `full` → 正常生成。

**定义规模层级后，通过 Task 使用 gate PR-SCOPE（`.claude/docs/director-gates.md`）生成 `producer`。**

传入：完整愿景规模、MVP 定义、时间线估算、团队规模。

向用户呈现评估。如果是 UNREALISTIC，在写文档之前提供调整 MVP 定义或规模层级的机会。

4. **使用模板生成游戏概念文档**，模板位于 `.claude/docs/templates/game-concept.md`。从构思对话中填写所有部分，包括 MDA 分析、玩家动机档案和心流状态设计部分。

   **在游戏概念文档中加入视觉身份锚点部分**，包含：
   - 选定的视觉方向名称
   - 一句话视觉规则
   - 2-3 个支撑视觉原则及其设计测试
   - 色彩哲学概要

   这个部分是艺术圣经的种子——它捕捉了"一切皆动"的决定，避免在会话之间被遗忘。

5. 使用 `AskUserQuestion` 请求写入批准：
- Prompt："游戏概念已就绪。可以写入 `design/gdd/game-concept.md` 吗？"
- Options：`[A] 是 — 写入` / `[B] 还没有 — 先修订某个部分`

如果选 [B]：用 `AskUserQuestion` 问要修订哪个部分：`Elevator Pitch` / `Core Fantasy & Unique Hook` / `Pillars` / `Core Loop` / `MVP Definition` / `Scope Tiers` / `Risks` / `其他 — 我来描述`

修订后，以 diff 或清晰的之前/之后形式展示更新后的部分，然后用 `AskUserQuestion`——"准备写入更新后的概念文档了吗？"
Options：`[A] 是 — 写入` / `[B] 再修订一个部分`
重复直到用户选择 [A]。

如果是，用 `.claude/docs/templates/game-concept.md` 的模板生成文档，从构思对话中填写所有部分，然后写入文件，必要时创建目录。

**规模一致性规则**："核心身份"表中的"预估规模"字段必须与规模层级部分中的完整愿景时间线一致——不只是说"大型（9+ 个月）"。要写成"大型（X-Y 个月，独立开发）"或"大型（X-Y 个月，N 人团队）"，这样摘要表才准确。

6. **建议下一步**（按此顺序——这是专业工作室的前期制作管线）。列出所有步骤——不要缩写或截断：
   1. "运行 `/setup-engine` 配置引擎并填充版本感知参考文档"
   2. "运行 `/art-bible` 创建视觉身份规范——这要在写 GDD **之前**做。艺术圣经制约资产生产并塑造技术架构决策（渲染、VFX、UI 系统）。"
   3. "使用 `/design-review design/gdd/game-concept.md` 验证概念完整性，再进入下游"
   4. "与 `creative-director` agent 讨论愿景以细化支柱"
   5. "用 `/map-systems` 将概念分解为独立系统——映射依赖关系、分配优先级并创建系统索引"
   5. "用 `/design-system` 为每个系统创作 GDD——按已识别系统的引导式、逐节 GDD 写作"
   6. "用 `/create-architecture` 规划技术架构——产出主架构蓝图和所需 ADR 列表"
   7. "用 `/architecture-decision (×N)` 记录关键架构决策——根据 `/create-architecture` 的所需 ADR 列表，每个决策写一个 ADR"
   8. "用 `/gate-check` 验证就绪状态——在投入生产前进行阶段门禁"
   9. "用 `/prototype [core-mechanic]` 原型化风险最高的系统——在全量实现之前验证核心循环"
   10. "原型完成后运行 `/playtest-report` 验证核心假设"
   11. "如果验证通过，用 `/sprint-plan new` 规划第一个冲刺"

7. **输出摘要**，包含选定概念的电梯演讲、支柱、主要玩家类型、引擎推荐、最大风险和文件路径。

1. `/setup-engine` — 配置引擎并填充版本感知参考文档
2. `/art-bible` — 在写任何 GDD 之前建立视觉身份
3. `/map-systems` — 将概念分解为有依赖关系的独立系统
4. `/design-system [first-system]` — 按依赖顺序为每个系统创作 GDD
5. `/create-architecture` — 产出主架构蓝图
6. `/gate-check pre-production` — 在投入生产前验证就绪状态
